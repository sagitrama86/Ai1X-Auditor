1. Problem Statement & Solution Design

1.1 The Problem

Healthcare organizations report HEDIS (Healthcare Effectiveness Data and Information Set) quality measures to NCQA. These measures are computed from a Reporting Data Warehouse (RDW) and displayed via Power BI reports. The validation challenge has three layers:

LayerWhat It RepresentsValidation Question
Layer 1: HEDIS StandardWhat NCQA says the measures SHOULD be"Is the client's implementation correct per HEDIS specs?"
Layer 2: RDW ImplementationWhat the client's database ACTUALLY contains"Does the SQL query produce the right numbers?"
Layer 3: Power BI ReportWhat the end user SEES on screen"Does the report match the database?"

Currently, validation is done manually: a human writes a 400+ line SQL query, runs it, exports Power BI data, and compares cell by cell in Excel. This process takes 2-3 days per report and is error-prone.

1.2 The Solution

Core Value Proposition: An AI-powered independent audit engine that combines HEDIS domain intelligence, database schema understanding, and LLM-powered natural language interaction to auto-generate validation SQL, detect data discrepancies, and identify gaps in report requirements — in minutes, not days.
  • HEDIS Domain Intelligence — Built-in knowledge of measures, clinical status codes, percentile calculations, and cross-field rules. The tool IS the HEDIS expert.
  • Database Schema Intelligence — Deep understanding of every table, column, join, and data pattern. Not just metadata — the tool knows what clinicalonlystatus=16 means.
  • AI-Powered Report Creation — User describes a report in one sentence; the tool generates complete metadata, SQL, validations, and identifies gaps.
  • Automated Comparison — Cell-by-cell diff between Power BI output and SQL query results with configurable tolerance.

1.3 Design Principles

  1. Skills-Based Architecture — Every piece of knowledge is a reusable YAML file. The tool gets smarter with each report defined.
  2. AI-First, Rule-Backed — LLM handles natural language; rule engine ensures correctness. The hybrid is the product.
  3. Independent Validation — Does NOT rely on client's requirements. Uses HEDIS specs as source of truth.
  4. Conversation as Authoring — Complex knowledge is taught via chat. Every conversation becomes a persisted skill.
  5. Zero Database Required for POC — All intelligence comes from YAMLs.

2.3 Technology Stack

Design Decision: Why YAML over a Database for Skills?
Skills are YAML files on disk because: (1) Human-readable — SMEs can review/modify directly. (2) Version-controllable via Git. (3) No infrastructure dependency. (4) Portable — copy the folder to deploy. Trade-off: No concurrent write safety. Acceptable for POC.

3.1.3 Performance Metric Invariants

These rules are ALWAYS true and auto-generated as cross-field validations:
  • Compliant + Non Compliant = Denominator
  • Compliant ≤ Denominator
  • Non Compliant ≤ Denominator
  • 0 ≤ Compliance % ≤ 100 (when Denominator > 0)
  • Compliance % = NULL when Denominator = 0
  • Required Exclusion members are NOT in Denominator

InvertedNCQAInvertedFlag = 'Y'ROUND(SUM(NotMet) * 100 / NULLIF(SUM(Met)+SUM(NotMet), 0), 2)NonCompliant as numerator

Buckets: <25th, 25th, 50th, 75th, 90th. NULL when: Denominator=0 or all cutpoints=0.

5. AI Engine — Deep Dive

5.1 System Prompt Construction

The system prompt is dynamically built from loaded skills and contains 11 components:

  1. All table names from schema
  2. All HEDIS measure codes
  3. All field template names
  4. Allowed data types: String, Integer, Decimal, Percentage, Date, Datetime
  5. Report type patterns (typical group_by + columns per type)
  6. All 11 standard slicer names
  7. First 5 cross-field rules
  8. Fixed filters
  9. 7 critical instructions (e.g., "group_by must include all non-agg fields")
  10. One complete few-shot example (correct MeasureList JSON)
  11. Output instruction: "Respond ONLY with valid JSON"

5.3 JSON Extraction

Handles 3 LLM response formats: (1) Clean JSON, (2) Markdown code block ```json...```, (3) JSON with surrounding text. Falls back to rule-based if all fail.

5.4 Fallback Engine

When Ollama is unavailable/times out (45s)/returns bad JSON: _rule_based_create_json() matches field templates by substring, then passes through the same 10-step validation. Output is structurally identical to LLM path.

5.5 Report Saving

On "yes save": generates report_id from name, builds YAML matching measurelist.yaml format, writes to skills/reports/, clears buffer, returns confirmation.

6. SQL Generation Engine

6.1 10-Step Pipeline

  1. Load report YAML → fields, joins, filters, dedup config
  2. Load HEDIS foundation → field templates
  3. Load join YAMLs → SQL for each join
  4. Build SELECT: for each field, look up template, expand complex references
  5. Add ROW_NUMBER for deduplication
  6. Build FROM + JOINs from join YAMLs
  7. Build WHERE: fixed filters + slicer values
  8. Build final SELECT from CTE
  9. Add WHERE RN = 1
  10. Add ORDER BY

6.2 Complex Field Expansion

ReferenceExpands ToLines

percentile_calculationNested CASE: NULL checks → NCQAInverted branch → 5 cutpoint comparisons~20

7. Compare Engine

7.1 Algorithm

  1. Parse — pd.read_csv(StringIO(csv)) for both files
  2. Normalize — Strip whitespace from column names
  3. Merge — pd.merge(on=composite_key, how='outer', indicator=True)
  4. Classify rows — left_only (extra in PBI), right_only (missing from PBI), both (matched)
  5. Diff cells — For each matched row × each value column:
    • Both NULL → MATCH
    • One NULL → MISMATCH
    • Both numeric → abs(a-b) ≤ tolerance → MATCH/MISMATCH
    • Both string → lower(trim(a)) == lower(trim(b)) → MATCH/MISMATCH
  6. Aggregate — match_rate = (total - mismatches) / total × 100
  7. Persist — Save run as JSON to runs/

7.2 Tolerance Handling

Data TypeMethodDefaultExample
Integerabs(a-b) ≤ 0.01Effectively exact1247 vs 1248 → MISMATCH
Decimalabs(a-b) ≤ 0.01±0.0185.43 vs 85.44 → MATCH
StringCase-insensitive, trimmed—"50th" vs "75th" → MISMATCH
NULLNULL == NULL—Both NULL → MATCH

7.3 Run History

Each run saved as runs/{YYYYMMDD_HHMMSS}.json with: run_id, timestamp, composite_key, tolerance, row counts, match_rate, full mismatch list. Retrieved via GET /api/runs (last 50, newest first).

11. Algorithms

11.1 Report Type Detection

lower = user_input.lower()
if "idss" in lower → "IDSSHierarchy"
elif "provider" in lower → "ProviderDetail"
elif "gap" in lower → "GapsInCare"
else → "MeasureList"

11.2 Field Template Matching

For each template_name in field_templates:
    if template_name.lower() in user_input.lower() → match
    elif any(word in user_input.lower() for word in template_name.split()) → match

For each standard_rule in HEDIS foundation:
    Extract field names from rule text
    If ALL referenced fields exist in report's field list → include rule
